Here's the entire JWT authentication flow condensed into **10 clear steps**:

1. **User logs in** by sending **email/username + password**. The server verifies the credentials against MongoDB.

2. If valid, the server generates an **Access Token** (short-lived, e.g., 15 minutes) and a **Refresh Token** (long-lived, e.g., 7 days). Both are JWTs signed with different secrets.

3. The **Refresh Token is stored in MongoDB** (`user.refreshToken`) and **both tokens are sent to the browser** as secure, HTTP-only cookies.

4. For every protected API request, the browser automatically sends the **Access Token**. The `verifyJWT` middleware reads it from the cookie or `Authorization` header.

5. `jwt.verify(accessToken, ACCESS_TOKEN_SECRET)` checks **(a)** the token was signed by your server, **(b)** it hasn't been tampered with, and **(c)** it hasn't expired. It then returns the payload (e.g., `_id`).

6. `verifyJWT` uses that `_id` to fetch the user from MongoDB, attaches it as `req.user`, and calls `next()`. Controllers can now use `req.user` without asking the client who they are.

7. When the **Access Token expires**, protected requests return **401 Unauthorized**. The user is **not logged out** yet.

8. The frontend automatically calls the **Refresh Token API**. The server verifies the Refresh Token using `REFRESH_TOKEN_SECRET`, fetches the user, and checks that the incoming refresh token **matches the one stored in MongoDB**.

9. If valid, the server **rotates the refresh token**: it generates a **new Access Token and a new Refresh Token**, stores the new Refresh Token in MongoDB, sends both back to the browser, and the old Refresh Token becomes invalid.

10. The user is forced to log in again only if they **explicitly log out** (refresh token removed from the database and cookies cleared), the **Refresh Token expires**, or the Refresh Token sent by the client **doesn't match** the one stored in MongoDB (stolen/old/reused token). This is what actually ends the session.
